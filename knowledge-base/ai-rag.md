# AI Integration & RAG Pipeline Experience

## My AI Experience
At Ingenio, I built a production RAG pipeline for the Keen platform — an advisor search system
that matches users seeking psychic, tarot, and spiritual guidance with the most relevant advisors.
The system handles all advisor profiles and combines semantic vector search with a structured
attribute scoring engine to rank results in real time.

I built this end-to-end in C# / .NET 8, integrating Azure OpenAI, PostgreSQL with pgvector,
and a custom hybrid ranking algorithm.

---

## What is RAG — How I Explain It
RAG stands for Retrieval-Augmented Generation. Instead of relying purely on an LLM's training data,
you first retrieve relevant context from your own data store, then pass that context to the LLM
so it answers based on YOUR data — not just what it was trained on.

In the Keen search system, the "retrieval" step is a vector similarity search across pre-computed
advisor profile embeddings stored in PostgreSQL. The "generation" step is an optional per-advisor
AI match summary generated by GPT-4.1-nano explaining why that advisor fits the user's need.

## How I Built the Keen Advisor Search RAG Pipeline

### Step 1 — Pre-compute Advisor Embeddings (Offline)
Each of the advisor profiles is embedded across 6 dimensions and stored in a
`advisor_profile_embeddings` table in PostgreSQL using the pgvector extension:
- `embedding_semantic_description` — Full profile meaning
- `embedding_keywords` — Specific topic relevance
- `embedding_expertise` — Domain expertise
- `embedding_tags`, `embedding_abilities`, `embedding_tools`

Why pre-compute? Computing embeddings for all advisors at query time would require
1,000+ API calls per search. Pre-computing means only 1 API call per search (for the user's query).

### Step 2 — At Query Time: Embed the User's Query
The user's search query (e.g. "I need a tarot reading about my love life") is sent to
Azure OpenAI's `text-embedding-3-small` model, which returns a `float[1536]` vector.
This is wrapped in a `Pgvector.Vector` object for proper Npgsql type mapping.

### Step 3 — pgvector Similarity Search (Multi-Embedding)
A single SQL query runs cosine similarity (`<=>` operator) against 3 embedding columns
simultaneously, with weighted scoring:
- `embedding_semantic_description`: 50% weight — overall profile meaning
- `embedding_keywords`: 30% weight — specific topic relevance
- `embedding_expertise`: 20% weight — domain expertise match

The formula: `1 - (embedding <=> query_vector)` converts cosine distance to similarity score
(1.0 = perfect match, 0.0 = no match). HNSW indexes on all 3 columns make this fast.

### Step 4 — Attribute Scoring Engine (7 Factors, No AI)
In parallel, a structured scoring engine evaluates each advisor on 7 factors — pure logic,
no AI, runs synchronously with zero latency:

| Factor | Weight | How It Works |
|--------|--------|--------------|
| Keywords | 25% | Query words matched against advisor's keyword list |
| Categories | 20% | Bidirectional containment match on expertise categories |
| Style | 15% | Exact match on communication style (e.g. "Compassionate") |
| Methods | 10% | Bidirectional match on tools/methods (e.g. "Tarot Cards") |
| Satisfaction | 10% | Normalized satisfaction score (0–100 → 0.0–1.0) |
| Experience | 10% | Mapped from data confidence: high=90, medium=70, low=50 |
| Empathy | 10% | Exact match = 1.0, any other = 0.5 partial credit |

### Step 5 — Hybrid Ranking
Final score combines both components:
```
Final Score = (Semantic Score × 0.40) + (Attribute Score × 0.60)
```
Results are sorted descending by final score and the top N are returned (default 20, max 500).

### Step 6 — AI Match Summary (Lazy, Per-Advisor)
A separate endpoint (`POST /api/match/summary`) generates a 2-sentence explanation
of why a specific advisor matches the user's query. Uses `ing-gpt-4.1-nano` (fast,
non-reasoning model) for ~200ms latency. This is called by the frontend per-advisor
as the user scrolls — NOT during the main search — to keep initial search fast.

## Challenges I Faced & How I Solved Them

### Challenge 1 — Multi-Embedding Strategy
**Problem:** A single embedding per advisor profile wasn't nuanced enough. A query about
"tarot love reading" needed to match keyword relevance, domain expertise, AND overall
profile meaning differently.

**Solution:** Pre-computed 6 separate embedding columns per advisor (semantic description,
keywords, expertise, tags, abilities, tools). At query time, 3 of them are queried
simultaneously with weighted scoring (50/30/20), giving far more nuanced retrieval
than a single embedding.

### Challenge 2 — Latency vs. Quality Trade-off
**Problem:** Running AI summary generation for all 20 search results would add ~4 seconds
of latency (200ms × 20 advisors).

**Solution:** Decoupled the summary into a separate endpoint (`POST /api/match/summary`)
called lazily by the frontend per-advisor as the user interacts with results. Main search
stays at 300–800ms.

### Challenge 3 — Circuit Breaker for Azure OpenAI
**Problem:** If Azure OpenAI goes down or is slow, the entire search would fail or hang.

**Solution:** Built a simple circuit breaker directly in `AzureOpenAIService`:
- Each embedding call has a **4-second timeout**
- After **2 consecutive failures**, the circuit opens for **60 seconds**
- While open, all calls return `float[0]` immediately and the search falls back
  to **attribute-only scoring** (60% of the engine still works)
- Users still get relevant results — they just lose the semantic component temporarily

### Challenge 4 — pgvector Type Mapping
**Problem:** Passing raw float arrays to pgvector queries failed or gave unreliable results.

**Solution:** Used `Pgvector.Vector` wrapper type with `dataSourceBuilder.UseVector()`
registration to ensure Npgsql correctly maps the `VECTOR(1536)` column type.

## Graceful Degradation Design

| Component Down | User Experience |
|----------------|-----------------|
| Azure OpenAI embeddings | Results ranked by 7-factor attribute scoring only — still relevant |
| Azure OpenAI summaries | Structured match reasons shown instead of AI explanation |
| pgvector | Falls back to attribute-only scoring |
| PostgreSQL | 500 error — complete failure |

## Why RAG Over Fine-Tuning — My View
For the Keen advisor search use case, RAG was the right choice because:
- Advisor data changes frequently (new advisors, updated profiles) — RAG updates without retraining
- We needed explainability — the attribute scoring engine provides structured match reasons
- Cost-effective — 1 embedding API call per search vs. expensive model training
- Semantic search gracefully degrades — the attribute engine still works if AI is unavailable
